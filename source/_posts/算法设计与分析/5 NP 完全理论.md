> 这篇笔记整理算法设计与分析中 NP 完全理论的主线。重点不是背一串困难问题，而是看清“验证容易、搜索困难”的问题如何通过多项式时间归约被放进同一个难度类。

## 从判定问题开始

复杂性理论通常先把优化问题改写成判定问题。比如旅行商问题可以写成：给定图、边权和阈值 `B`，是否存在一条总长度不超过 `B` 的哈密顿回路。这样做的好处是可以统一讨论“输入是否属于 Yes 实例集合”。

`P` 表示可以被确定型算法在多项式时间内判定的问题。`NP` 表示给定一个 certificate 以后，可以在多项式时间内验证答案为 Yes 的问题。这里的 `NP` 是 nondeterministic polynomial，不是 non-polynomial。

因此：

- 如果一个问题在 `P` 中，它一定在 `NP` 中，因为可以忽略 certificate 直接求解。
- 对很多组合问题，我们知道它们在 `NP` 中，却不知道是否在 `P` 中。
- `P = NP` 仍然是公开问题，通常被认为不成立。

## 多项式时间归约

归约写作 `A <=p B`，意思是：存在一个多项式时间可计算的变换 `f`，使得 `x` 是 `A` 的 Yes 实例，当且仅当 `f(x)` 是 `B` 的 Yes 实例。

直觉上，`B` 至少和 `A` 一样难。因为如果能快速求解 `B`，就可以先把 `A` 的输入变成 `B` 的输入，再调用 `B` 的算法来解 `A`。

复习时最容易错的是归约方向：要证明一个新问题 `X` 很难，应当从一个已知 NP 完全问题归约到 `X`，也就是写 `Known-NPC <=p X`。

## NP 难与 NP 完全

一个问题是 NP-hard，表示所有 `NP` 中的问题都能多项式时间归约到它。一个问题是 NP-complete，则还要同时满足它自己也在 `NP` 中。

证明一个问题 `X` 是 NP 完全，标准模板只有两步：

1. 证明 `X in NP`：给出 certificate 和多项式时间验证算法。
2. 选一个已知 NP 完全问题 `Y`，证明 `Y <=p X`。

如果某个 NP-hard 问题能被放进 `P`，那么所有 `NP` 问题都能通过归约被多项式时间求解，进而得到 `P = NP`。这也是 NP 完全理论的实际价值：它告诉我们何时应当停止寻找精确多项式算法，转向近似、启发式、特殊情形或指数算法。

## Cook-Levin 定理

Cook-Levin 定理给出第一个 NP 完全问题：SAT 是 NP 完全的。

SAT 在 `NP` 中很直接：给定一个布尔赋值，只要线性扫描公式就能验证它是否满足所有子句。NP-hard 的关键思想是：任意 `NP` 问题都有一个多项式时间验证器，可以把“存在一条接受计算路径”编码成一个布尔公式。

这个编码通常称为 tableau。它用二维表记录非确定型图灵机每一步、每个纸带格子的状态，再用 CNF 子句约束：

- 每个格子每个时刻只能有一种符号或状态。
- 第 0 行必须是输入对应的初始配置。
- 某一行必须进入接受状态。
- 相邻两行之间必须符合转移函数。

公式可满足，当且仅当存在一条合法接受路径。因此任意 `NP` 问题都能归约到 SAT，SAT 就是 NP-hard；再加上 SAT 在 `NP` 中，所以 SAT 是 NP 完全的。

## 经典归约链

有了 SAT 以后，可以建立一串常见归约：

```text
SAT -> 3-SAT -> Independent Set -> Vertex Cover -> Clique
                     |
                     v
              Hamiltonian Cycle -> TSP
```

3-SAT 到 Independent Set 的构造最典型：每个子句中的每个文字建一个点，同一子句内的点互斥，互为否定的文字之间也连边。能选出 `m` 个两两不冲突的点，当且仅当原公式的 `m` 个子句可以同时满足。

Independent Set、Vertex Cover 与 Clique 之间的关系也常考：

- 图中大小为 `k` 的 independent set，对应补图中大小为 `k` 的 clique。
- 图中大小为 `k` 的 vertex cover，对应原图中大小为 `|V| - k` 的 independent set。

TSP 的判定版本常从 Hamiltonian Cycle 归约而来：原图中存在的边赋权 1，不存在的边赋权 2，阈值设为 `|V|`。只有当原图有哈密顿回路时，TSP 才能走出总长度不超过 `|V|` 的巡回。

## 复习时抓住什么

NP 完全理论的复习重点是证明结构，而不是孤立结论：

- 先明确问题是否是判定版本。
- 证明 `in NP` 时写 certificate 和验证复杂度。
- 证明 NP-hard 时先选已知困难问题，再保证归约方向正确。
- 写归约时同时说明构造时间和 Yes/No 等价性。

遇到新问题时，可以先问它像 SAT、图选择问题、路径问题、装包问题中的哪一类。找到相似结构以后，再考虑用 gadget 或补图关系把“选择且不冲突”“覆盖所有约束”“存在一条合法路径”这类条件转译过去。
