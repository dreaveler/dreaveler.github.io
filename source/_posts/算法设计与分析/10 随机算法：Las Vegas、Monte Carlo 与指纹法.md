> 这篇笔记整理随机算法的基本范式：为什么给算法“投硬币”可以避开确定性最坏输入，Las Vegas 和 Monte Carlo 分别保证什么，以及随机快排、随机选择、主元素测试、串相等与 Rabin-Karp 怎样体现这些思想。

## 为什么需要随机化

确定性算法的选择规则一旦固定，就可能被输入“针对”。快速排序如果每次选第一个元素做枢轴，在已经排好序的数组上会退化到平方时间。随机算法把某些选择交给随机数，使输入本身不能决定算法的整条执行路径。

随机化并不是放弃分析。它把复杂度或正确性写成关于随机选择的概率命题：

- 对任意固定输入，运行时间的期望是多少。
- 对任意固定输入，输出错误的概率至多是多少。
- 通过重复独立运行，失败概率怎样指数下降。

这也是随机算法和“平均输入分析”的区别。随机快排的输入可以是任意坏输入，期望是对算法内部硬币求的。

## Las Vegas 与 Monte Carlo

随机算法常分成两类。

Las Vegas 算法的答案一定正确，随机性只影响运行时间。它可能偶尔慢，也可能一次尝试失败后重来，但只要返回一个结果，这个结果就是合法答案。

Monte Carlo 算法的运行时间通常有确定上界，但输出可能出错。它关心的是错误概率，比如单次错误率不超过 `1/2` 或 `1/n^2`。如果每次运行独立，并且单次成功概率至少为 `p`，重复 `k` 次后全部失败的概率至多为

```text
(1 - p)^k
```

当 `p = 1/2` 时，重复 `k` 次失败率就是 `2^-k`。这叫 boost：用线性增加的运行次数换指数下降的错误概率。

两类算法可以在特殊条件下互相转化。如果 Monte Carlo 的错误是单边的，并且候选答案能廉价验证，就可以在验证失败时重跑，从而得到 Las Vegas 算法。

## 随机快排：正确性不变，时间变成期望保证

随机快速排序只改一个地方：每次划分时，从当前数组中均匀随机选一个枢轴。划分、递归和普通快排一样。

算法仍然总是输出正确排序，所以它是 Las Vegas 算法。随机性只影响递归树是否平衡。

设输入有 `n` 个互异元素。每次划分要做 `n - 1` 次比较，枢轴落在第 `i` 小位置的概率是 `1/n`。期望比较次数满足

```text
T(n) = n - 1 + (1/n) * sum_{i=0}^{n-1} [T(i) + T(n-1-i)]
     = n - 1 + (2/n) * sum_{i=0}^{n-1} T(i)
```

由归纳或积分上界可以得到

```text
T(n) <= 2n ln n
```

更直观的证明是指示变量法。把排序后序列记作 `z_1 < z_2 < ... < z_n`，令 `X_ij` 表示 `z_i` 和 `z_j` 是否被比较。两者会比较，当且仅当区间 `{z_i, ..., z_j}` 中第一个被选为枢轴的是两端之一。因此

```text
P[X_ij = 1] = 2 / (j - i + 1)
```

总比较次数是所有 `X_ij` 的和。利用线性期望，可以推出期望比较次数是 `O(n log n)`。注意最坏情况下它仍可能是 `Theta(n^2)`，只是这种极端选择序列的概率很低。

## 随机选择：期望线性找第 k 小

随机选择问题是：在数组中找第 `k` 小元素。它和快排类似，只是每次划分后只递归到包含第 `k` 小元素的一边。

流程是：

1. 随机选一个枢轴。
2. 把数组划分成小于、等于、大于枢轴的三部分。
3. 若第 `k` 小落在左边，只递归左边；若落在右边，调整 `k` 后递归右边；若等于枢轴，直接返回。

它也是 Las Vegas 算法：答案一定正确，运行时间取决于枢轴质量。粗略上界中，每次划分成本为 `O(n)`，而递归只进入一侧；随机枢轴使期望递归规模足够小，最终得到期望时间 `O(n)`。

与确定性“中位数的中位数”算法相比，随机选择没有最坏线性保证，但实现简单、常数小，在工程中更常用。

## n 后问题中的随机尝试

n 后问题要求在 `n x n` 棋盘上放 `n` 个皇后，使任意两个皇后不同行、不同列、不同对角线。

一种 Las Vegas 思路是逐行放置。第 `k` 行先列出所有与前面皇后不冲突的列，再随机选一个。如果某一行没有候选列，本次尝试失败；如果成功放完 `n` 行，就得到正确解。

这个算法的关键不是“每次都能成功”，而是“成功时一定正确”。失败可以重来。若单次成功概率为 `p`，期望尝试次数是 `1/p`。也可以先随机放前若干行，再用确定性回溯补全，把随机搜索和系统搜索结合起来。

## Monte Carlo 主元素测试

数组中的主元素是出现次数超过 `n/2` 的元素。确定性做法可以用 Boyer-Moore 投票线性求出。随机做法更简单：

1. 随机抽一个位置。
2. 统计该元素出现次数。
3. 若超过 `n/2`，报告它是主元素；否则报告失败。

如果主元素存在，一次抽中它的概率超过 `1/2`。如果抽到并统计通过，答案必然正确。重复 `k` 次仍未发现主元素的概率小于 `2^-k`。这类算法常见特点是：不会把错误元素误报成主元素，但可能漏报存在的主元素，因此错误是单边的。

## 指纹法：用小随机素数比较长串

串相等测试的场景是：两方各有一个很长的二进制串，想用很少通信判断它们是否相等。

直接传输整串当然正确，但成本太高。指纹法把长串看作大整数 `x` 和 `y`，随机选一个小素数 `p`，只比较

```text
x mod p  与  y mod p
```

如果 `x = y`，模任何 `p` 都相等。如果 `x != y`，只有当 `p` 整除 `|x - y|` 时才会误判。随机从足够大的素数集合中选 `p`，误判概率可以压得很低。

指纹法体现了 Monte Carlo 的典型权衡：用一个短摘要代替完整对象，换取低概率错误。它的错误通常也是单边的：不相等可能被误判为相等，相等不会被误判为不等。

## Rabin-Karp：滚动指纹做模式匹配

Rabin-Karp 模式匹配要在文本 `T` 中找模式串 `P`。朴素算法会逐个位置比较 `P` 和窗口 `T[i..i+m-1]`，最坏时间可能是 `O(mn)`。

Rabin-Karp 的思想是给每个长度为 `m` 的窗口算哈希指纹：

```text
hash(T[i..i+m-1])
```

如果窗口哈希和 `P` 的哈希不同，窗口一定不匹配；如果哈希相同，再逐字符验证。为了在线性时间处理所有窗口，哈希必须能滚动更新：从当前位置移动到下一个位置时，删掉最高位字符，加入新字符，常数时间得到新哈希。

在合适的随机模数下，假阳性概率很低。算法的实际运行接近 `O(n + m)`，并且通过“哈希命中后再验证”可以避免输出错误匹配。

## 该记住的判断标准

遇到随机算法时，先问三件事：

1. 随机性影响的是时间还是正确性？这决定它更像 Las Vegas 还是 Monte Carlo。
2. 概率是对输入求的，还是对算法内部随机选择求的？随机算法分析通常固定输入，只对硬币求期望或错误率。
3. 错误能否验证？如果能，Monte Carlo 常可以改造成 Las Vegas；如果不能，就要用重复独立运行把错误率降到可接受水平。

随机化的价值不在“凭运气”，而在把对手最坏输入的结构打散，并给出可证明的期望性能或高概率保证。
